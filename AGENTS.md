# Agent / assistant notes

- **Contributor workflow and commands:** [CONTRIBUTING.md](CONTRIBUTING.md), [docs/DEVELOPER.md](docs/DEVELOPER.md) (includes **VS Code / gopls** `fullruntime` + **`gopls_stub`** for terrain stubs on Windows and **`scripts/check_builds.sh`**), [docs/PHYSICS.md](docs/PHYSICS.md) (Dual-path architecture), and [docs/STATE_OF_THE_UNION.md](docs/STATE_OF_THE_UNION.md) (Current Engine Status).
- **Default `go run .` only compiles to `.mbc`.** Graphical programs need **`-tags fullruntime`** and **`moonrun`** or **`moonbasic --run`** (built with fullruntime).
- **Manifest changes:** edit `compiler/builtinmanifest/commands.json`, then run `go run . --check` on a relevant sample; regenerate `docs/API_CONSISTENCY.md` with `go run ./tools/apidoc` when the public API surface changes.
- **Easy Mode & Handle Methods (Consistency Rule):** Prioritize using handle methods (e.g., `hero.SetGravity(1.0)`) over global namespaced commands (e.g., `CHARACTERREF.SETGRAVITY`) in all premium showcase examples and Mario 64 parity scripts. Ensures a unified developer path.
- **Hardware Abstraction Layer (HAL):** See **[`docs/architecture/HAL_AND_RENDERING.md`](docs/architecture/HAL_AND_RENDERING.md)** for the full picture (null vs Raylib drivers, injection, Windows purego deferred DLL load, static-link experiments).
  - **Prefer** `rt.Driver` and `hal` types for **new** window/video/input code and for refactors that touch the HAL slice.
  - **Migration**: Many `runtime/*` packages still import `raylib-go` directly; do not assume the tree is already driver-pure—extend HAL incrementally.
  - **Headless first**: Compiler paths (`ListBuiltins`, `NewRegistryHeadless`) and null-driver tests must stay runnable without a GPU or sidecar **`raylib.dll`** where the build tag matrix allows it.
- **Windows first, Linux second (docs + workflow):** Prefer **Windows** as the primary dev surface and put **Windows** before **Linux** in reference tables and platform notes ([docs/DEVELOPER.md](docs/DEVELOPER.md#platform-priority-windows-then-linux)). **Linux** remains essential for full **Jolt** validation and Unix CI.
- **Windows and Linux (code):** Register new builtins on every split build path (`*_cgo.go` / `*_stub.go` or equivalent) so both OSes expose the same manifest keys—use stubs or no-ops where native code is platform-specific. Run `bash scripts/check_builds.sh` on Linux/macOS; on Windows use the same `go build` tag matrix from [docs/DEVELOPER.md](docs/DEVELOPER.md).
- **`runtime/player`:** Desktop **Windows/Linux** uses a single **Jolt-backed** implementation (`(linux || windows) && cgo`); **`!cgo`** builds get stubs with an explicit **CGO + Jolt** error string—there is no second “host” KCC solver in Go.
- **Physics sync & Jolt (keep this in mind on every change):** See [docs/PHYSICS.md](docs/PHYSICS.md) for dual-path sync, vendored **`third_party/jolt-go`** limitations (`JOINT.*` placeholders, `APPLYFORCE` = impulse×dt, `SetRaycastHook`), and contributor rules.
  - **`entity_phys_sync_cgo.go`**, **`entity_jolt_collision_cgo.go`**, and related **mbentity** hooks use **`//go:build cgo || (windows && !cgo)`** so **one implementation ships on Windows and Linux** fullruntime builds—do **not** fork these behind `linux`-only tags.
  - **Native Jolt (desktop)** is gated **`//go:build (linux || windows) && cgo`** in files such as [`runtime/physics3d/jolt_state_cgo.go`](runtime/physics3d/jolt_state_cgo.go), [`pick_cgo.go`](runtime/physics3d/pick_cgo.go), [`collision_bridge_cgo.go`](runtime/physics3d/collision_bridge_cgo.go). **Stub** companions use the complement **`(!linux && !windows) || !cgo`**—not `!linux || !cgo` (that wrongly includes stubs on **Windows+CGO** and causes duplicate symbols). See [docs/PHYSICS.md](docs/PHYSICS.md#build-tag-contract-for-physics3d).
  - **`ENTITY.ADDPHYSICS` / `ENTITY.PHYSICS`**: real bodies when **`(linux || windows) && cgo`** ([`entity_physics_macro_cgo.go`](runtime/mbentity/entity_physics_macro_cgo.go)); **`entity_physics_macro_stub.go`** otherwise so the **same `.mb` compiles** everywhere.
  - Ground probe + visual Y snap in **`syncEntitiesFromPhysics`** only affect **Jolt-linked** entities when the probe succeeds; stub builds never see hits, so behavior degrades safely.
  - **`ENTITY.SETBOUNCE` / `ENTITY.SETFRICTION`**: must update **Jolt** restitution/friction when **`physBufIndex >= 0`** (`SetRestitutionToIndex` / `SetFrictionToIndex`), not only `e.bounce` / `e.friction`—otherwise `player.SetBounciness(0)` and floors never match **`PHYSICS.BOUNCE`** behavior.
  - **Solver slop (~0.02 m)** and micro-bounce: **`syncEntitiesFromPhysics`** widens the visual snap band slightly and, when grounded, zeros **tiny** vertical linear velocity (`|vy|` below a small threshold) while keeping **XZ**—reduces ping-pong with the raycast-grounded path without killing jumps (large `vy`).
